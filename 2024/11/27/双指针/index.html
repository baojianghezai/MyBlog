双指针

2024/11/27 算法

洛谷-P1147 连续自然数和

先来看一道例题

先来理解一下题意：给出一个整数M，构建一个从1到M的数列，然后从数列中选出一个l和r，如果l+(l+1)+(l+2)+……+r = M 的话，则输出l和r

正常思路：从0到M枚举l，再去找出是否存在r值满足上面条件

代码如下：

#include <iostream>
using namespace std;

void su( int l , int r ) //求出l到r所有数的和
{
    int suma = 0;
    for( int i = l; i <= r ; i++ )
    {
        suma += i;
    }
    return suma;
}

int main()
{
    int m;
    cin>>m;
    for( int l = 1 ; l <= m ; l++ )
    {
        for( int r = l ; r <= m ; r++ )
        {
            if( su(l,r) == m )
            {
                cout<<l<<" "<<r<<endl;
            }
        }
    }
}

嗯~再去瞅一眼时间复杂度

两层for循环，N^2了，再加上su函数也算一个l到r的循环

因此……时间复杂度来到了恐怖的N^3

M的数据量在10~2e6之间，所以这道题用这种方法的话包超时的

再想想怎么去优化

首先想的的是我真的需要循环这么多次数吗？

由题目中描述“给出一个满足条件的连续正整数段中的第一个数和最后一个数”，可以知道两个数必然是连续的，又想到在M/2往后将不会满足条件→当l=M/2的时候，下一个就已经超过M/2，因此循环可以减少一半

再去想，当我的某个l找到对应的r之后，还需要继续去移动我的r指针吗？

→很明显答案是不需要，因为这时候r再+1的话，结果必定会大于M

因此我们可以直接break掉

修改之后的代码

#include <iostream>
using namespace std;

void su( int l , int r ) //求出l到r所有数的和
{
    int suma = 0;
    for( int i = l; i <= r ; i++ )
    {
        suma += i;
    }
    return suma;
}

int main()
{
    int m;
    cin>>m;
    for( int l = 1 ; l <= m/2 ; l++ )
    {
        for( int r = l ; r <= m/2 ; r++ )
        {
            if( su(l,r) == m )
            {
                cout<<l<<" "<<r<<endl;
                break;
            }
        }
    }
}

时间复杂度还是很大，这时候又可以想到从l加到r可以用一个叫前缀和的东西去解决

因此构建前缀和数组，优化函数部分

#include <iostream>
using namespace std;

const int N = 2e6+10;
int a[N];

int main()
{
    int m;
    cin>>m;
    for( int i = 1 ; i <= m ; i++ )  //构建前缀和数组
    {
        a[i] = a[i-1] + i;
    }
    for( int l = 1 ; l <= m/2 ; l++ )
    {
        for( int r = l ; r <= m/2 ; r++ )
        {
            if( a[r]-a[l-1] == m )
            {
                cout<<l<<" "<<r<<endl;
                break;
            }
        }
    }
}

好，时间复杂度已经优化到N^2/4 了

但是还是非常大

这时候再去想：

枚举l的时候，是否需要将r移动到l+1的地方

推断一下：用sum表示从l加到r，假设此时sum==m，当l往右移动一个的时候，sum变为sum-l，此时如果将r向右移动的话，必然会导致最终的sum>m，这时候只能将l再次向右移动，但移动到什么程度为止，很明显我们不能非常容易的得到答案

那么我们可以去换一种思路

我将sum和m状态分为三种

sum > m

sum = m

sum < m

在这三种状态下，再去想需要进行什么操作

sum > m 需要去减小sum，要么l向右，要么r向左，由于我们需要将指针向一个方向移动，所以需要将l向右移动→既l++

sum = m 记录并输出

sum < m 需要增大sum，l向左移动，或者r向右移动，和第一个一样，选择向右移动→既r++

那这时候可能就会有杠精站出来了：

要是l向右移动的时候，漏掉了正确答案呢（也就是r需要向左动去寻找答案）

那这里咱也可以去推断一下嗷

首先l初始为1，r根据判断条件，无非两种结果：要么找到了sum=m的点，要么直接变成sum > m ，接下来就是去移动l了，当l向右移动过程中，如果有合适的r值，那么肯定是和l一起连续移动，那么一定会与现在的r值重合或者掠过，重合就是sum=m，略过就是sum>m，需要再去移动r，因此可以知道l向右移动的时候一定不会出现略过正确答案的情况

因此代码最终形态就变成了这样

#include <iostream>
using namespace std;

const int N = 2e6+10;
int a[N];

int main()
{
    int m;
    cin>>m;
    for( int i = 1 ; i <= m ; i++ )
    {
        a[i] = a[i-1] + i;
    }
    for( int l = 1 , r = 1 ; l <= m/2 + 1 ;)
    {
        int x = a[r] - a[l-1];
        while( x < m )
        {
            r++;
            x = a[r] - a[l-1];
        }
        while( x > m )
        {
            l++;
            x = a[r] - a[l-1];
        }
        if( x == m )
        {
            cout<<l<<" "<<r<<endl;
            l++;
            continue;
        }
        if( r > m || l > m/2 ) break;
    }
}

//但是由于这道题数据卡的不是很严格，所以下面这种做法也可以

#include <iostream>
using namespace std;
const int N = 1e6 + 10;
int a[N],b[N];
int main()
{
    int m;
    cin>>m;
    for( int i = 1 ; i <= m/2 + 1 ; i++ )
    {
        a[i] = i;
        b[i] = b[i-1] + a[i];
    }
    for( int l = 1 ; l <= m/2 + 1 ; l++ )
    {
        for( int r = l+1 ; r <= m/2 + 1 ; r++ )
        {
            int x = b[r] - b[l-1];
            if( x > m ) break;
            if( x == m )
            {
                cout<<l<<" "<<r<<endl;
                break;
            }
        }
    }
    return 0;
}

这种思路就是当我r找的时候，如果发现sum>m，那么就没有必要寻找下去了，直接break掉将l向右移动一个

这个时间复杂度的话，假设r做多找到m/2，那么就是m^2/4，其实随着l越来越大，r需要找的就更少，因此时间应该是远少于这个值的，具体看测试点吧，个人不太推荐这种做法
